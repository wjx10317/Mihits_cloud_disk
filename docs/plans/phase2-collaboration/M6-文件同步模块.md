# M6 文件同步模块

## 模块概述

文件同步模块实现本地与云端的文件自动同步，支持双向/单向同步模式、增量同步、冲突检测与处理、定时同步和离线缓存。这是网盘客户端区别于普通文件管理器的核心能力。

## 技术选型

| 层面 | 技术 |
|------|------|
| 文件监听 | chokidar（Node.js 原生文件监听库） |
| 文件指纹 | SHA-256（分块哈希，避免全文件读取） |
| 差异比对 | 基于时间戳 + 文件大小 + 哈希三重判断 |
| 冲突策略 | 用户选择：覆盖 / 跳过 / 保留两者 |
| 定时任务 | node-cron |
| 增量同步 | 记录 sync_cursor（同步游标），仅同步游标之后的变更 |

## 数据库设计

```sql
-- 同步目录配置
CREATE TABLE sync_folders (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    local_path VARCHAR(1024) NOT NULL,       -- 本地目录路径
    cloud_folder_id UUID REFERENCES folders(id) ON DELETE CASCADE,  -- 云端目录
    sync_mode VARCHAR(20) NOT NULL DEFAULT 'bidirectional',  -- bidirectional/upload_only/download_only
    is_active BOOLEAN DEFAULT TRUE,
    last_sync_at TIMESTAMP WITH TIME ZONE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- 同步记录（用于增量同步）
CREATE TABLE sync_records (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    sync_folder_id UUID NOT NULL REFERENCES sync_folders(id) ON DELETE CASCADE,
    file_path VARCHAR(1024) NOT NULL,
    cloud_file_id UUID REFERENCES files(id) ON DELETE SET NULL,
    local_hash VARCHAR(64),
    cloud_hash VARCHAR(64),
    local_modified_at TIMESTAMP WITH TIME ZONE,
    cloud_modified_at TIMESTAMP WITH TIME ZONE,
    sync_status VARCHAR(20) DEFAULT 'synced',  -- synced/pending/conflict/error
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- 同步冲突记录
CREATE TABLE sync_conflicts (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    sync_folder_id UUID NOT NULL REFERENCES sync_folders(id) ON DELETE CASCADE,
    file_path VARCHAR(1024) NOT NULL,
    conflict_type VARCHAR(50) NOT NULL,       -- both_modified/local_deleted_cloud_modified/cloud_deleted_local_modified
    local_version_hash VARCHAR(64),
    cloud_version_hash VARCHAR(64),
    resolution VARCHAR(20),                    -- overwrite_local/overwrite_cloud/keep_both/skip
    resolved_at TIMESTAMP WITH TIME ZONE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- 离线操作日志
CREATE TABLE offline_operations (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    operation_type VARCHAR(50) NOT NULL,      -- create/update/delete/rename/move
    file_path VARCHAR(1024),
    file_data BYTEA,                          -- 小文件数据缓存
    metadata JSONB,                           -- 操作元数据
    synced BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

## 子模块详细设计

### M6.1 同步设置

**前端**：
- 设置页「同步」标签页
- 功能：
  - 添加同步目录：选择本地目录 + 选择云端目录
  - 同步模式切换：双向同步 / 仅上传 / 仅下载
  - 同步目录列表：显示各目录状态、上次同步时间
  - 启用/停用某个同步目录
  - 移除同步目录

**后端**：
- `POST /api/v1/sync/folders` — 添加同步目录
- `GET /api/v1/sync/folders` — 获取同步目录列表
- `PUT /api/v1/sync/folders/{id}` — 更新同步配置
- `DELETE /api/v1/sync/folders/{id}` — 移除同步目录
- `POST /api/v1/sync/folders/{id}/trigger` — 手动触发同步

### M6.2 本地监听

**前端（Electron 主进程）**：
- 使用 chokidar 监听同步目录
- 事件类型：`add` / `change` / `unlink` / `addDir` / `unlinkDir`
- 防抖处理：文件修改后等待 1s 无新变更再触发同步
- 忽略规则：
  - 隐藏文件（`.`开头）
  - 临时文件（`~`开头、`.tmp`后缀）
  - 系统文件（`Thumbs.db`、`.DS_Store`）
  - 超大文件提示确认

```typescript
class SyncWatcher {
  private watchers: Map<string, FSWatcher> = new Map()
  
  startWatching(syncFolder: SyncFolder) {
    const watcher = chokidar.watch(syncFolder.localPath, {
      ignored: /(^|[\/\\])\..|\.tmp$|~$|Thumbs\.db|\.DS_Store/,
      persistent: true,
      ignoreInitial: true,        // 忽略初始扫描
      awaitWriteFinish: {
        stabilityThreshold: 1000, // 1s 稳定期
        pollInterval: 200
      }
    })
    
    watcher.on('all', (event, path) => {
      this.handleLocalChange(syncFolder, event, path)
    })
    
    this.watchers.set(syncFolder.id, watcher)
  }
}
```

### M6.3 增量同步

**同步策略**（三重判断）：

1. **时间戳比对**：本地修改时间 vs 云端修改时间
2. **大小比对**：文件大小是否变化
3. **哈希比对**：仅在前两者都变化时计算（节省性能）

**双向同步流程**：
```
1. 拉取云端变更列表（since last_sync_at）
2. 扫描本地变更（chokidar 事件队列）
3. 比对双方变更：
   a. 仅本地变更 → 上传
   b. 仅云端变更 → 下载
   c. 双方均有变更 → 进入冲突处理
   d. 均无变更 → 跳过
4. 执行同步操作
5. 更新 sync_records
6. 更新 last_sync_at
```

**后端接口**：
- `GET /api/v1/sync/changes?since={timestamp}&folder_id={id}` — 获取云端变更
- `POST /api/v1/sync/report` — 上报本地变更，服务端比对返回操作指令
```json
{
  "sync_folder_id": "uuid",
  "local_changes": [
    { "path": "/docs/file.txt", "action": "modified", "hash": "sha256...", "size": 1024 }
  ]
}
```
- 响应：
```json
{
  "code": "SUCCESS",
  "data": {
    "actions": [
      { "file": "docs/file.txt", "action": "upload", "reason": "local_newer" },
      { "file": "docs/readme.md", "action": "download", "reason": "cloud_newer" },
      { "file": "docs/notes.txt", "action": "conflict", "reason": "both_modified" }
    ]
  }
}
```

### M6.4 冲突处理

**前端**：
- 冲突对话框，逐个或批量处理：
  - 覆盖本地版本（以云端为准）
  - 覆盖云端版本（以本地为准）
  - 保留两者（冲突文件重命名，如 `file_conflict_20260429.txt`）
  - 跳过（不处理，下次同步再次提醒）

**自动冲突策略**（可配置）：
- 默认：弹窗让用户选择
- 可选：总是保留两者 / 总是以云端为准 / 总是以本地为准

**后端**：
- `POST /api/v1/sync/conflicts/{id}/resolve`
```json
{ "resolution": "overwrite_local|overwrite_cloud|keep_both|skip" }
```

### M6.5 定时同步

**前端**：
- 同步设置中配置定时同步规则
- 支持间隔：每 5 分钟 / 15 分钟 / 30 分钟 / 1 小时 / 自定义
- 开机自启同步开关

**实现**：
- Electron 主进程使用 node-cron 定时触发
- 开机自启通过 electron-store 记录配置
- 同步进行中不重复触发（锁机制）

### M6.6 离线缓存

**前端**：
- 网络断开时，所有文件操作记录到 offline_operations（IndexedDB）
- 操作类型：创建、修改、删除、重命名、移动
- 小文件（< 1MB）同时缓存文件数据
- 网络恢复后自动执行离线操作队列

**后端**：
- `POST /api/v1/sync/offline/batch` — 批量提交离线操作
- 校验每个操作的可行性（如删除的文件在云端也被删除则跳过）
- 返回每个操作的执行结果

## 错误码

| 错误码 | HTTP状态 | 说明 |
|--------|----------|------|
| SYNC_FOLDER_NOT_FOUND | 404 | 同步目录不存在 |
| SYNC_IN_PROGRESS | 409 | 同步正在进行中 |
| LOCAL_PATH_INVALID | 400 | 本地路径无效 |
| LOCAL_PATH_DUPLICATE | 409 | 本地路径已被其他同步目录使用 |
| CONFLICT_NOT_FOUND | 404 | 冲突记录不存在 |

- 关键测试场景：
  - 本地文件变更自动检测
  - 双向同步正确性
  - 增量同步（仅同步变更部分）
  - 冲突检测与解决
  - 离线操作队列
  - 网络恢复后自动同步
  - 定时同步触发
  - 同步中重复触发防护
